## Architecture Patterns

### GoReleaser v2 Library Mode Config

**What:** GoReleaser v2 configuration for pure library (no binary) distribution.

**When to use:** Library modules published to `pkg.go.dev` and GitHub Releases.

**Correct structure:**
```yaml
# .goreleaser.yaml
version: 2

project_name: openclaw-sdk-go

before:
  hooks:
    - go mod tidy
    # No go generate (no directives exist)
    # No go test (CI already validates)

gomod:
  proxy: true   # Enables gomod.proxy support
  mod: mod

builds:
  - id: go-build
    skip: true  # Library -- no binaries

snapshot:
  version_template: "{{ incpatch .Version }}-next"

release:
  github:
    owner: frisbee-ai
    name: openclaw-sdk-go
  mode: github      # Publish to GitHub Releases (NOT replace)
  prerelease: auto
  draft: false
```

**Anti-patterns to Avoid:**
- `mode: replace` for libraries -- replaces existing release instead of creating new GitHub Release
- `blobs: true` for libraries -- meant for binary distribution with external storage (not needed here)
- Including `go test ./...` in `before.hooks` -- CI already runs tests; wastes release time
- Including `go generate ./...` when no generate directives exist -- no-op that adds latency

### git-cliff Integration Pattern

**What:** git-cliff generates `--release-notes` for GoReleaser from conventional commits.

**Integration point:** `.github/workflows/release.yml` runs:
```bash
git-cliff --config cliff.toml --strip all -- ${PREV_TAG}..${CURRENT_TAG} > /tmp/RELEASE_NOTES.md
goreleaser release --clean --release-notes /tmp/RELEASE_NOTES.md
```

**cliff.toml coverage:** Already configured for all commit types used in project:
- `feat` -> Features
- `fix` -> Bug Fixes
- `perf` -> Performance
- `refactor` -> Refactor
- `test` -> Testing
- `doc` -> Documentation
- `build`, `ci`, `chore` -> Miscellaneous/Build

## Common Pitfalls

### Pitfall 1: Wrong GoReleaser Mode for Libraries

**What goes wrong:** `mode: replace` in GoReleaser v2 replaces the existing GitHub Release instead of creating a new one. Each version overwrites the previous release notes and artifacts.

**Why it happens:** `mode: replace` is the default behavior but inappropriate for library distribution where you want release history preserved.

**How to avoid:** Use `mode: github` which creates a new GitHub Release for each tag.

**Warning signs:** GitHub Releases page shows only one release with latest tag, previous release notes overwritten.

### Pitfall 2: Missing GitHub Release Section with `mode: github`

**What goes wrong:** GoReleaser fails with "invalid configuration: missing GitHub release configuration" when `mode: github` is set but `release.github` section is absent.

**Why it happens:** `mode: github` requires explicit `github` section with `owner` and `name`.

**How to avoid:** Add the required section:
```yaml
release:
  github:
    owner: frisbee-ai
    name: openclaw-sdk-go
```

### Pitfall 3: Including Unnecessary Hooks

**What goes wrong:** `go test ./...` in `before.hooks` runs tests during release, which is redundant since CI already validates tests and slows down the release process.

**Why it happens:** Copying CI steps into release hooks without recognizing duplication.

**How to avoid:** Only keep `go mod tidy` in hooks. Remove `go test` and `go generate` unless specifically needed.

### Pitfall 4: Tag Placement After Previous Release

**What goes wrong:** Creating `v1.0.0` tag on a later commit when `v1.0.0` already exists at an earlier commit causes duplicate tag errors or confusion.

**Why it happens:** Tags are global in git -- once a tag name is used, it cannot be recreated on a different commit.

**How to avoid:** Use `v1.0.1` for the next release since `v1.0.0` is already placed.
