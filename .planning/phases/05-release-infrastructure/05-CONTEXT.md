# Phase 5: Release Infrastructure - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 5 delivers **library distribution ready for v1.0 release** for the OpenClaw SDK. Three requirements:

- **REL-01**: GoReleaser library mode — `blobs: true` and `gomod.proxy: true`; release builds without error
- **REL-02**: Semantic version tags exist — `v0.1.0`, `v0.2.0`, ... progression leading to `v1.0.0`
- **REL-03**: Changelog automation configured — git-cliff generates CHANGELOG.md from conventional commits

</domain>

<decisions>
## Implementation Decisions

### REL-01: GoReleaser Configuration

- **D-01 (Release mode):** Use `mode: github` — publish GitHub Releases with proper release notes. NOT `mode: replace`.
- **D-02 (before.hooks):** Keep `go mod tidy`. Remove or skip `go test ./...` since CI already validates tests. Keep `go generate ./...` only if the project has generate directives; otherwise remove it.
- **D-03 (gomod.proxy):** Already configured as `proxy: true`. This enables `gomod.proxy: true` for Go 1.20+ module proxy distribution.
- **D-04 (blobs/distribution):** Library mode — no binaries to distribute. GoReleaser should push to GitHub Releases only. No `blobs` storage needed.
- **D-05 (release.draft):** `draft: false` — releases go live immediately.

### REL-02: Semantic Versioning

- **D-06 (Existing tag):** `v1.0.0` tag already exists at commit `3e27f32`. Subsequent ~9 commits (Phase 3 + Phase 4 work) exist on top. Cannot re-create `v1.0.0`.
- **D-07 (Next version strategy):** Since `v1.0.0` is taken at an earlier commit, the next release should be `v1.0.1` (patch bump). No need for v0.x.y progression — the library is production-ready.
- **D-08 (Version template):** Use GoReleaser default semver template: `{{ .Tag }}`. No custom template needed.

### REL-03: Changelog Automation

- **D-09 (git-cliff):** Already fully configured. `cliff.toml` exists at repo root with:
  - Conventional commit parsing (feat, fix, perf, refactor, test, doc, etc.)
  - Categorized groups with emojis (Features, Bug Fixes, Performance, etc.)
  - Breaking change detection
  - Full changelog and release notes generation
- **D-10 (CI integration):** `.github/workflows/release.yml` already integrates git-cliff with GoReleaser. No changes needed.
- **D-11 (CHANGELOG.md update):** After release, CHANGELOG.md should be updated with the new release section via git-cliff. Consider adding a pre-commit hook to update CHANGELOG.md on each commit.

### REL-04: CI Release Job

- **D-12 (release.yml):** `.github/workflows/release.yml` already exists and is well-structured. It:
  - Triggers on `tags: ['v*.*.*']` push
  - Installs git-cliff via taiki-e/install-action
  - Determines previous tag for changelog generation (handles both stable and prerelease tags)
  - Runs goreleaser with `--clean --release-notes` flags
  - Uses `fetch-depth: 0` for full git history
  - No changes needed to release workflow.

</decisions>
